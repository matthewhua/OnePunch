# 早期开发阶段单点部署方案 (Single World Server Guide)

在游戏开发的早期阶段（Alpha 或本地测试期），使用**单台 World 服（单节点）**不仅完全可行，而且是强烈推荐的最佳实践。

## 1. 为什么推荐早期使用单节点？

> [!TIP]
> 聚焦核心业务逻辑，晚点再去硬碰硬处理分布式难题。

- **调试极简**：所有大地图状态都在同一个进程的内存里，可以直接通过 IDE 的 Debug 模式加断点，追踪玩家从发兵到战斗的全生命周期，无需跨进程分析日志追踪。
- **避免分布式强一致性开销**：当跨区域行军或战斗发生时，不需要真实的 gRPC 网络传输或分布式锁，降低了早期的业务复杂度。
- **环境搭建快**：客户端只需直连一个服务网关，并代理至唯一的一个 World 服务端进程；开发人员在自己的一台笔记本电脑上即可跑全套系统。

## 2. 现有架构如何完美兼容单节点？

我们在 [Stage 10] 中设计的 `Router` (一致性哈希路由) 并不会因为只配置了一个节点而出现问题，它能天生自适应：

```rust
let mut router = Router::new();
// 早期开发：只灌入单台服务器
router.add_node("Local_World_01");

// 无论任何坐标 (x,y)
let target = router.route_by_coord(150, 150).unwrap();
assert_eq!(target, "Local_World_01"); 
// 路由层会自动命中这一台机器，相当于直连。
```

从逻辑上，这使得您的架构在一开始就是完备...94条代码中。

## 3. 后期如何无缝向集群演讲 (Scale Out)？

由于我们提前准备好了 `Sector` 和 `Actor` 系统，单节点到多节点的切换几乎算是**降维打击般的平滑**。

当游戏的 DAU (日活) 上升，单台物理机的 CPU（主要是行军定时器 tick 和战斗结算）和内存吃满时：

1. **修改网关/Router配置**：将启动参数从 `["Local_World_01"]` 变更为 `["Node_A", "Node_B", "Node_C", "Node_D", "Node_E"]`。
2. **多进程部署**：在运维环境打包出 5 个一样的 World Server 进程。
3. **Actor 分区自动切割**：虽然是不同的进程，但各自的 `MapSectorActor` 会按需被唤醒。Router 根据坐标把东边的交战请求投递给 Node_A，西边的投递给 Node_B。

## 4. 结论

**放心大胆地用单服进行开发。** 

您不用去掉目前写的这些健壮性和路由的代码模块。把他们留着，只是在配置文件中将集群列表填成 **长度为 1 的数组** 即可。保持 “逻辑上切分，物理上聚合” 的架构。
